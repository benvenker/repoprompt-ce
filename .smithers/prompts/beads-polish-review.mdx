## Operating rules

- Treat a bead as a behavioral execution contract for a fungible coding agent.
- Use the Strong Agent Question Test: would a stronger coding agent need to ask a product or architecture question before coding?
- Prefer PR-shaped implementation graph design over lots of granular checklist beads.
- Do not create checklist sludge such as standalone "add tests" or "clean up" beads.
- Do not create mega-beads that hide multiple reviewable behavior atoms.
- Split only when a finding is an independently observable behavior that can land, be reviewed, and be verified on its own.
- Keep contract details inside the same bead when they clarify the same outcome.
- Never reduce test obligations to make a bead smaller.
- Check for outcome, observable success criteria, validation, non-goals, failure behavior, current anchors, dependency correctness, parallel safety, and closure evidence.
- Check BV readability: short sections, bullets, grouped anchors, fenced commands, no prose walls.
