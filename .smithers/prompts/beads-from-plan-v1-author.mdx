## Mutation rules

- Use `br` for all Beads mutations.
- Use `br --json` and `bv --robot-*` for inspection.
- Never edit `.beads/issues.jsonl` directly.
- Never run bare `bv`.
- Do not implement product code.
- Every created bead must include `--external-ref` set to the plan path above.
- Every created bead must include the lane label above.
- Use `br dep add <child> <parent>` for real dependency/order edges.
- Use `br update` to repair existing plan-linked beads.
- If beads already exist for this plan, do not duplicate them; update/split/merge
  only when the local workflow rubric says the graph shape needs it.

## Graph decomposition mode

Apply this section only when the shape gate says `graph_decomposition`.
Otherwise skip it, except for the compact split decision requested by the
single-bead fast path.

When graph decomposition is selected, design a PR-shaped implementation graph:

- Parent / PR-slice beads are closure contracts.
- Child beads are reviewable behavior atoms, not checklist crumbs.
- Preserve all final product and architecture decisions from the plan.
- Include outcome, success criteria, scope/non-goals, failure behavior,
  validation, anchors/surfaces, dependency/parallel notes, and closure evidence.
- Include behavior/contract/regression test intent where appropriate.
- Mark only the true ready frontier with `ready-for-agent`.

Run a split test before mutating:

- List the plan's behavior atoms in your own notes.
- Split when two atoms can land as separately reviewable behavior/system truths,
  or when one bead would combine multiple high-diff-risk dimensions such as
  service behavior, runtime surface wiring, response-shape preservation,
  smoke-harness work, docs/status closeout, or a broad new test harness.
- Do not split checklist crumbs such as "add tests" or "update docs" unless the
  item closes a separate observable system truth.
- A single bead remains acceptable when the atoms all prove one PR-sized
  outcome. In that case, include only a compact "Split decision" or equivalent
  note explaining why no parent/child graph is needed.
- If you create children, create a parent closure bead and represent order with
  `br dep add <child> <parent>` plus any child-to-child ordering edges needed.

## Single-bead fast path

If the shape gate says `single_bead_fast_path`, do not carry graph-decomposition
scaffolding into the bead. Create or repair exactly one compact implementation
bead unless existing inventory proves a different repair is necessary.

Ignore the graph decomposition section above, including behavior-atom notes,
parent/child rationale, graph analysis, and dependency notes. The only graph
shape artifact to keep is a compact split decision.

For fast-path beads, keep the description tight and execution-oriented. Prefer
these sections only:

- Outcome
- Must do
- Must not do / STOP
- Validation
- Closure evidence

Treat docs/status updates as closure evidence for the same bead.
