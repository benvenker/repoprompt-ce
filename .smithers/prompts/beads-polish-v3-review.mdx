## Operating rules

- Before reviewing, read `.smithers/skills/beads-polish-v3.md` and
  `.smithers/skills/beads-from-plan-v1.md`; apply their Strong Agent Question
  Test, split/merge discipline, quality-gate rules, and BV readability guidance.
- Treat a bead as a behavioral execution contract for a fungible coding agent.
- Use the Strong Agent Question Test: would a stronger coding agent need to ask a product or architecture question before coding?
- Prefer PR-shaped implementation graph design over lots of granular checklist beads.
- Do not create checklist sludge such as standalone "add tests" or "clean up" beads.
- Do not create mega-beads that hide multiple reviewable behavior atoms.
- Split only when a finding is an independently observable behavior that can land, be reviewed, and be verified on its own.
- Keep contract details inside the same bead when they clarify the same outcome.
- Never reduce test obligations to make a bead smaller.
- Run an explicit split decision for the bead under review:
  identify behavior atoms, decide which are same-outcome details versus
  independently reviewable system truths, and say whether the current bead can
  remain one reviewable implementation bead.
- A dense single bead is acceptable only when it is a small sub-plan or child
  outcome and the bead itself explains why no parent/child graph is needed. If
  that explanation is absent or unconvincing, recommend a graph correction.
- Treat a missing split decision as a quality issue even when the bead passes
  deterministic checks; do not reward compact prose that hides graph shape.
- Check for outcome, observable success criteria, validation, non-goals, failure behavior, current anchors, dependency correctness, parallel safety, and closure evidence.
- Check BV readability: short sections, bullets, grouped anchors, fenced commands, no prose walls.
- If previous verifier feedback names missing context or context sweep requests,
  inspect that context before recommending a mutation. Use targeted commands such
  as `br show --json`, `br dep ... --json`, `bv --robot-*`, `rg`, and focused file
  reads. Do not convert verifier feedback into wording changes until you have
  enough current context to know what behavior, anchor, failure mode, or
  validation evidence is actually missing.
