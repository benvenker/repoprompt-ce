## Context sweep

Inspect current code/docs enough to name:

- user-visible surfaces and runtime paths touched by the plan;
- likely implementation seams, files, commands, or existing patterns;
- relevant data contracts, state transitions, or process lifecycle behavior;
- failure behavior the implementation beads must preserve;
- focused validation commands that should appear in the beads.

Prefer `rg` and targeted file reads. Do not overfit exact file lists into edit
scripts; treat them as anchors future agents should verify.

## Graph-shape split test

Before proposing a graph shape, identify the plan's behavior atoms. Classify
each atom as one of:

- contract detail for the same reviewable implementation outcome;
- independent behavior/system truth that can land and be reviewed separately;
- closure/status/docs evidence that belongs in a parent or final child.

Default to a parent closure bead plus child implementation beads when the plan
contains multiple independently reviewable outcomes or high-diff-risk
dimensions. A single directly implementable bead is allowed only when the plan is
already a small sub-plan and every atom is necessary to prove the same PR-sized
behavior. If you recommend one bead, explicitly say why it is not hiding a
parent/child graph.

## Output contract

Return a structured brief:

- `planPath`
- `laneLabel`
- `planSummary`
- `finalDecisions`
- `proposedGraphShape`
- `behaviorAtoms`
- `singleBeadJustification` when proposing one directly implementable bead
- `contextSweep`
- `anchors`
- `validationPlan`
- `risks`
- `authoringInstructions`

The authoring instructions should explicitly tell the next agent how to create
self-contained Beads using `br create`, `br update`, and `br dep add`, with:

- `--external-ref` set to: {props.planPath}
- labels including: {props.laneLabel}
- parent closure contracts for PR/lane slices;
- child implementation beads that are reviewable behavior atoms;
- `ready-for-agent` only on the real frontier.
